# PLAN - WP-2026-173

## Resumen
WP-2026-173 agrega un helper proactivo `--pre-handoff` en `agent_controller.py` para que Builder llegue a `--mark-ready` con commit y checkpoint M3 ya preparados.

## Problema
- Hoy el guard detecta el error al final, pero el Builder sigue pudiendo llegar a `--mark-ready` sin commit ni tag alineado.
- El flujo manual de commit + checkpoint + verificacion limpia se repite demasiado y es facil de saltar.

## Objetivo
1. Añadir un comando unico `--pre-handoff`.
2. Encadenar commit, checkpoint M3 y verificacion limpia antes del handoff.

## Alcance
- `.agent/agent_controller.py`
- `tests/test_agent_controller.py`

## Decisiones
- El helper no modifica el guard ni `bus/supervisor.py`.
- El helper falla cerrado si no hay cambios de entrega, si no puede crear/renovar el checkpoint o si no queda arbol limpio.
- El comando lee `Files Likely Touched` desde `work_plan.md` con logica inline equivalente al parser del guard.
- El checkpoint M3 se crea o renueva inline con `git rev-parse checkpoint/review-<ticket>^{}` + `git tag -d` + `git tag -a`.
- El commit usa el mensaje estandar `chore(<ticket>): pre-handoff checkpoint`.
- Si `git commit` falla por hooks de pre-commit, el stderr real se expone tal cual al usuario.
- La comprobacion final de arbol limpio debe usar `git status --porcelain` y filtrar las mismas superficies vivas toleradas por el guard, incluyendo `TURN.md`, `STATE.md`, `execution_log.md` y `events.jsonl`.
- El comando debe dejar el workspace listo para `--mark-ready` sin pasos manuales adicionales.

## Criterios
- `--pre-handoff` crea o actualiza el commit y el checkpoint M3.
- `--pre-handoff` devuelve exito idempotente si no hay cambios de entrega y el checkpoint ya esta alineado.
- `--pre-handoff` deja el arbol limpio, ignorando superficies vivas toleradas, o devuelve error claro.
- Los tests cubren el camino feliz, la idempotencia sin cambios y los fallos de preflight.
