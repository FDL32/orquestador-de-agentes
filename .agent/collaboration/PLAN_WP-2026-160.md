# Work Plan - WP-2026-160

## Metadata
- **ID:** WP-2026-160
- **Estado:** IN_PLANNING
- **deliverable_type:** code
- **Titulo:** Restart supervisor on Builder relaunch
- **Asignado a:** Builder

## Objetivo
Hacer que el camino `-ResumeBuilder` garantice que el supervisor en memoria sea reemplazado por uno fresco antes de abrir el nuevo Builder, cerrando el anti-patron de procesos de larga vida con codigo viejo en disco.

## Contexto
- AP-13 ya documenta el problema: un proceso supervisor de larga vida puede seguir ejecutando codigo viejo despues de un hot-patch si no se reinicia.
- Hoy `scripts/launch_agent_terminals.ps1 -ResumeBuilder` se usa en el camino de requeue, pero no deja explicitado ni verificado que el supervisor que queda activo sea uno fresco.
- El objetivo de este ticket es arreglar la orquestacion de arranque sin inventar un daemon nuevo: el launcher coordina el reemplazo y el supervisor viejo sale limpiamente para liberar el lock.
- El fix debe mantener el flujo bus-first: el launcher coordina, el supervisor sigue siendo un consumidor reactivo y Builder no intenta repararse a si mismo.

## Decision Arquitectonica
- Opcion elegida: el supervisor viejo sale limpiamente tras el requeue, el launcher espera a que el lock desaparezca y luego arranca un supervisor fresco antes de abrir Builder.
- Secuencia de operaciones obligatoria: (1) `run_reactive()` sale del bucle tras un requeue exitoso, liberando `supervisor_lock.txt` via el bloque `finally` existente; (2) el launcher hace poll de `supervisor_lock.txt` hasta que desaparece o hasta un timeout; (3) el launcher arranca un supervisor fresco usando el mismo patron de arranque normal; (4) el launcher abre Builder. Si cualquier paso falla, el launcher escribe un mensaje especifico a stderr y sale con exit code no cero sin abrir Builder.
- La pieza en `bus/supervisor.py` es minima: en `run_reactive()`, despues de que `requeue_ticket()` devuelva `True`, romper el bucle y devolver. El bloque `finally` existente libera el lock sin cambios adicionales.
- El launcher NO elimina `supervisor_lock.txt` de forma directa; espera a que el supervisor viejo lo libere por si mismo tras su salida cooperativa. Esto evita la condicion de carrera donde el proceso viejo sigue vivo aunque el lock haya desaparecido.
- La trazabilidad del reinicio es stderr explicito desde el launcher mas un evento de bus `SUPERVISOR_RESTARTED` emitido por el supervisor nuevo al arrancar, con payload `{"round": N, "reason": "resume-builder"}`. No hay evento ad hoc desde el launcher.
- El reinicio del supervisor debe reutilizar el bloque de arranque que ya usa el camino normal del launcher, no un nuevo camino ad hoc.

## Non-goals
- No crear un daemon nuevo.
- No mover la responsabilidad de requeue al supervisor.
- No introducir polling continuo ni backoff de reintentos.
- No cambiar el contrato de `--mark-ready`.

## Fases

### Fase 0: instrumentacion del reinicio
- **Tipo:** TAREA AGENTE
- **Archivos:** `scripts/launch_agent_terminals.ps1`, `bus/supervisor.py`, `tests/test_launch_agent_terminals_script.py`, `tests/test_supervisor.py`
- **Accion:** Instrumentar
- **Descripcion:** Aislar el camino de refresco del supervisor con un helper observable, por ejemplo `Wait-SupervisorExit` o `Ensure-FreshSupervisor`, y anadir en `run_reactive()` de `bus/supervisor.py` la salida del bucle tras un requeue exitoso. El helper del launcher debe hacer poll de `supervisor_lock.txt` con timeout y dejar claro si el supervisor salio, si expiro el timeout o si el reinicio fallo.
- **Riesgo:** Medio
- **Criterio de Aceptacion:** Un test de contenido puede afirmar que `-ResumeBuilder` llama al helper de espera antes de abrir Builder y que `run_reactive()` rompe el bucle tras requeue exitoso. Un test de unidad del supervisor verifica que `run_reactive()` devuelve tras el primer requeue exitoso sin continuar el bucle.
- **Si falla:** Mantener la instrumentacion minima como mensaje/log visible y posponer la automatizacion total.

### Fase 1: fix dirigido de reinicio
- **Tipo:** TAREA AGENTE
- **Archivos:** `scripts/launch_agent_terminals.ps1`, `bus/supervisor.py`, `tests/test_launch_agent_terminals_script.py`, `tests/test_supervisor.py`
- **Accion:** Endurecer
- **Descripcion:** Implementar el camino de `-ResumeBuilder` siguiendo la secuencia: (1) `run_reactive()` sale del bucle tras requeue exitoso y libera el lock; (2) el helper del launcher hace poll de `supervisor_lock.txt` hasta que desaparece o vence el timeout; (3) el launcher arranca un supervisor fresco con el mismo patron normal; (4) el supervisor nuevo emite `SUPERVISOR_RESTARTED` con `{"round": N, "reason": "resume-builder"}`; (5) el launcher abre Builder. Si el poll vence sin que el lock desaparezca, el launcher escribe un mensaje especifico a stderr y sale con exit code no cero sin abrir Builder.
- **Riesgo:** Medio
- **Criterio de Aceptacion:** En el flujo `ResumeBuilder`, el supervisor viejo no sobrevive como autoridad activa cuando Builder vuelve a entrar; el camino de relanzado deja trazabilidad observable del reinicio, usa el mismo patron de arranque normal del launcher y no abre Builder si el supervisor no pudo renovarse.
- **Si falla:** Mantener el guard minimo de no abrir Builder si el supervisor no puede verificarse como fresco, con stderr explicito y exit code no cero.

### Fase 2: smoke path de requeue con supervisor fresco
- **Tipo:** TAREA AGENTE
- **Archivos:** `tests/test_launch_agent_terminals_script.py`, `tests/test_supervisor.py`
- **Accion:** Anadir cobertura
- **Descripcion:** Cubrir el flujo de requeue que usa `-ResumeBuilder` con tests de contenido para el launcher y tests de unidad para el supervisor: el launcher debe mostrar la rama explicita de reinicio y el supervisor debe exponer la salida cooperativa tras el handoff. La cobertura debe demostrar que el camino de requeue no depende de un supervisor viejo persistiendo en memoria.
- **Riesgo:** Bajo
- **Criterio de Aceptacion:** La prueba deja claro que `-ResumeBuilder` prepara un supervisor fresco antes del Builder nuevo, que el supervisor viejo sale limpiamente y que el camino de cierre no se apoya en un proceso viejo.
- **Si falla:** Conservar la cobertura de la instrumentacion y limitar el smoke al orden supervisor->builder con fail-closed observable.

- `-ResumeBuilder` deja explicitamente un supervisor fresco antes de abrir Builder.
- Un supervisor viejo no queda como autoridad activa despues del relanzado.
- El camino de reinicio es observable y falla cerrado si no puede garantizar frescura, con stderr explicito y exit code no cero.
- El supervisor viejo sale limpiamente tras el requeue y libera el lock sin intervencion del launcher.
- El supervisor nuevo emite `SUPERVISOR_RESTARTED` con `{"round": N, "reason": "resume-builder"}` al arrancar.
- La validacion canonica y la suite safe siguen pasando.
