# Work Plan - WP-2026-160

## Metadata
- **ID:** WP-2026-160
- **Estado:** IN_PLANNING
- **deliverable_type:** code
- **Titulo:** Restart supervisor on Builder relaunch
- **Asignado a:** Builder

## Objetivo
Hacer que el camino `-ResumeBuilder` del launcher garantice que el supervisor en memoria no quede obsoleto antes de abrir el nuevo Builder, para cerrar el anti-patron de procesos de larga vida con codigo viejo en disco.

## Contexto
- AP-13 ya documenta el problema: un proceso supervisor de larga vida puede seguir ejecutando codigo viejo despues de un hot-patch si no se reinicia.
- Hoy `scripts/launch_agent_terminals.ps1 -ResumeBuilder` se usa en el camino de requeue, pero no deja explicitado ni verificado que el supervisor que queda activo sea uno fresco.
- El objetivo de este ticket es arreglar la orquestacion de arranque, no introducir self-reload en `bus/supervisor.py`.
- El fix debe mantener el flujo bus-first: el launcher coordina, el supervisor sigue siendo un consumidor reactivo, y Builder no intenta repararse a si mismo.

## Decision Arquitectonica
- Opcion elegida: el launcher reinicia o reemplaza el supervisor al relanzar Builder.
- La comprobacion de frescura vive en `scripts/launch_agent_terminals.ps1`; no se introduce logica de auto-vigilancia en `bus/supervisor.py`.
- `-ResumeBuilder` debe tratar `supervisor_lock.txt` como estado antiguo del ciclo previo: si existe y corresponde a una sesion vieja, se limpia antes de relanzar Builder.
- El launcher debe dejar trazabilidad observable del reinicio del supervisor para que el Manager pueda decidir sin inferir.
- No se anade watcher, daemon ni reintentos incrementales: el ticket solo endurece el reinicio del supervisor en el camino de requeue.

## Non-goals
- No tocar `bus/supervisor.py` salvo que aparezca una evidencia real de bloqueo que no pueda resolverse en el launcher.
- No crear un daemon nuevo.
- No mover la responsabilidad de requeue al supervisor.
- No introducir polling continuo ni backoff de reintentos.
- No cambiar el contrato de `--mark-ready`.

## Fases

### Fase 0: instrumentacion del reinicio
- **Tipo:** TAREA AGENTE
- **Archivos:** `scripts/launch_agent_terminals.ps1`, `tests/test_launch_agent_terminals_script.py`
- **Accion:** Instrumentar
- **Descripcion:** Aislar el camino de refresco del supervisor en el launcher con un helper observable, por ejemplo `Restart-ProjectSupervisor` o `Ensure-FreshSupervisor`. Ese camino debe dejar claro si el supervisor se limpio, si ya estaba fresco o si el reinicio fallo. La salida del launcher debe permitir distinguir el estado sin leer la ventana manualmente.
- **Riesgo:** Medio
- **Criterio de Aceptacion:** Un test del launcher puede afirmar, leyendo el script o el helper expuesto, que `-ResumeBuilder` tiene una rama explicita para dejar el supervisor fresco antes de abrir Builder.
- **Si falla:** Mantener la instrumentacion minima como mensaje/log visible y posponer la automatizacion total.

### Fase 1: fix dirigido de reinicio
- **Tipo:** TAREA AGENTE
- **Archivos:** `scripts/launch_agent_terminals.ps1`, `tests/test_launch_agent_terminals_script.py`
- **Accion:** Endurecer
- **Descripcion:** Implementar el camino de `-ResumeBuilder` para que limpie o sustituya un `supervisor_lock.txt` stale y relance un supervisor fresco antes de abrir Builder. Si el supervisor no puede quedar fresco, el launcher debe fallar cerrado y no abrir Builder sobre un proceso viejo.
- **Riesgo:** Medio
- **Criterio de Aceptacion:** En el flujo `ResumeBuilder`, el supervisor viejo no sobrevive como autoridad activa cuando Builder vuelve a entrar; el camino de relanzado deja trazabilidad observable del reinicio y no abre Builder si el supervisor no pudo renovarse.
- **Si falla:** Mantener el guard minimo de no abrir Builder si el supervisor no puede verificarse como fresco.

### Fase 2: smoke path de requeue con supervisor fresco
- **Tipo:** TAREA AGENTE
- **Archivos:** `tests/test_launch_agent_terminals_script.py`
- **Accion:** Anadir cobertura
- **Descripcion:** Cubrir el flujo de requeue que usa `-ResumeBuilder` para verificar que el launcher pasa por el reinicio de supervisor antes de Builder y que el orden de operaciones no se invierte. La cobertura debe demostrar que el camino de requeue no depende de un supervisor viejo persistiendo en memoria.
- **Riesgo:** Bajo
- **Criterio de Aceptacion:** La prueba deja claro que `-ResumeBuilder` prepara un supervisor fresco antes del Builder nuevo y que el camino de cierre no se apoya en un proceso viejo.
- **Si falla:** Conservar la cobertura de la instrumentacion y limitar el smoke al orden supervisor->builder.
