# Audit - WP-2026-171

## Lo que ya existe
- `scripts/pre_handoff_guard.py` ya verifica checkpoint M3 y arbol limpio antes de permitir `--mark-ready`.
- `scripts/create_checkpoint.py` ya crea el tag `checkpoint/review-<ticket>` como checkpoint M3.
- `.agent/agent_controller.py` ya invoca el pre-handoff guard dentro de `--mark-ready`, pero el mensaje preventivo debe residir en el guard.
- `check_checkpoint_m3_exists()` debe refactorizarse en `check_checkpoint_alignment(project_root, ticket_id) -> tuple[bool, bool]` para distinguir checkpoint ausente de checkpoint desalineado.

## Blockers
- Si el guard solo comprueba que el tag exista, el Builder puede seguir llegando con un checkpoint desalineado respecto al commit de entrega.
- Si el guard no diferencia entre checkpoint ausente y checkpoint desalineado, el mensaje quedara ambiguo y el Builder no sabra si debe crear o recrear el tag.
- Si los tests solo cubren el checkpoint ausente y no el checkpoint desalineado, la nueva regla quedara incompleta.

## Evidencia esperada
- Un test que pruebe el rechazo cuando `checkpoint/review-<ticket>` no existe.
- Un test que pruebe el rechazo cuando el checkpoint existe pero no corresponde al HEAD entregable.
- Un test que pruebe el camino feliz con checkpoint y commit alineados.
- Un mensaje de rechazo claro para Builder indicando si debe crear el tag o recrearlo porque apunta a otro commit, con estas instrucciones exactas:
  - Tag ausente: `git commit && git tag -a checkpoint/review-<ticket> -m "Checkpoint M3 for <ticket>"`
  - Tag desalineado: `git tag -d checkpoint/review-<ticket> && git tag -a checkpoint/review-<ticket> -m "Checkpoint M3 for <ticket>"`
