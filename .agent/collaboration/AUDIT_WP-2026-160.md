# Audit - WP-2026-160

## Lo que ya existe
- `AP-13` ya documenta el patron de proceso de larga vida que sigue ejecutando codigo viejo tras un hot-patch.
- `scripts/launch_agent_terminals.ps1` ya tiene el camino `-ResumeBuilder`, la limpieza de locks y el arranque de ventanas.
- `tests/test_launch_agent_terminals_script.py` ya valida el contenido del launcher y puede servir como base de regresion.

## Blockers
- `-ResumeBuilder` no garantiza que el supervisor en memoria quede fresco antes de abrir Builder -> el ticket debe introducir una rama explicita de reinicio/renovacion del supervisor en el launcher.
- El camino de requeue puede seguir confiando en un supervisor viejo si no se limpia o reemplaza `supervisor_lock.txt` -> el ticket debe dejar el lock viejo fuera de juego antes de abrir Builder.
- No hay trazabilidad observable del reinicio del supervisor -> el ticket debe dejar un mensaje o helper verificable que permita al Manager comprobar el cambio por diff o por salida de script.
- Si el fix se filtra a `bus/supervisor.py`, el alcance se ha desviado -> el ticket debe mantenerse en launcher + tests salvo evidencia nueva.

## Evidencia esperada
- Un test del launcher que pruebe que `-ResumeBuilder` fuerza la frescura del supervisor antes del Builder.
- Un test que compruebe que el camino falla cerrado si no se puede garantizar un supervisor fresco.
- `pytest`, `pytest-safe` y `agent_controller --validate` verdes.
