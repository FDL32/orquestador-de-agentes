# AUDIT_WP-2026-146 - Human gate timeout and expiry

## Audit Focus

Verify that `HUMAN_GATE` is backed by the existing persistent approval subsystem so timeout expiry survives restarts and resolves through the current approval-resolution contract without introducing a new state-machine branch.

## What Must Be True

- The `STATE_CHANGED -> HUMAN_GATE` handoff includes timeout metadata.
- The escalation creates a persisted `ApprovalRequest` in `bus/approval.py`.
- The supervisor can expire the persisted request through its existing `check_and_expire_all()` loop.
- The existing `APPROVAL_RESOLVED` -> `BLOCKED` path remains the source of truth for timeout expiry.
- The timeout value is sourced from configuration with a documented fallback.
- `resume-human-gate` still works for a live gate and is not affected by the timeout logic.
- The implementation stays small and does not expand into a broader runtime refactor.

## Failure Modes

1. The gate remains open forever because no persisted approval exists to expire.
2. The ticket gets a new terminal state instead of using the existing approval resolution contract.
3. The timeout metadata is added but never consumed by the persistent approval path.
4. `resume-human-gate` is broken or becomes incompatible with a live gate.
5. The change spills into unrelated runtime projection work.

## Review Checklist

- Confirm the handoff carries timeout metadata.
- Confirm the escalation persists an `ApprovalRequest`.
- Confirm the supervisor expires only the intended gate, not fresh work, even after restart.
- Confirm the bus derives the expired path through the existing contract.
- Confirm the timeout value comes from configuration.
- Confirm the builder resume path still works after the change.
- Confirm tests cover expired and non-expired cases.
