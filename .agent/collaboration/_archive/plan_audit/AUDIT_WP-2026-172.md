# Audit - WP-2026-172

## Lo que ya existe
- `scripts/pre_handoff_guard.py` ya filtra varias superficies vivas del runtime.
- `bus/supervisor.py` ya emite `BUILDER_RELAUNCH_ATTEMPTED` y relanza Builder cuando detecta requeue.
- `HANDOFF_BLOCKED` ya se emite desde `agent_controller.py` cuando el guard bloquea el handoff.

## Riesgos
- Si `PROJECT.md` no entra en las superficies vivas, el guard puede seguir bloqueando un cierre que solo actualiza estado operativo.
- Si el supervisor no distingue un `HANDOFF_BLOCKED` posterior al trigger de requeue de crash/timeout, puede crear una segunda instancia de Builder sobre el mismo arbol.

## Evidencia esperada
- Un test del guard donde `PROJECT.md` cambie y el handoff siga siendo valido.
- Un test del supervisor donde `HANDOFF_BLOCKED` posterior al trigger evita relanzado.
- Un test del supervisor donde timeout o ausencia de actividad relevante sigue permitiendo relanzado.
