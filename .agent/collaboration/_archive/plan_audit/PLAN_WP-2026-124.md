# PLAN_WP-2026-124 - Drift canonico del bus

## Estrategia

WP-2026-124 cierra el drift entre el bus canonico y las proyecciones humanas.
La solucion adoptada es la opcion (b): la materializacion canonica vive en
`agent_controller.py` y el bridge del Manager la dispara por CLI, igual que ya
hace para `changes`. Asi se evita cualquier import circular entre
`bus/review_bridge.py` y `agent_controller.py`.

## Enfoque tecnico

1. Reutilizar la materializacion existente en `agent_controller.py` via una
   ruta CLI nueva para `inspect` (`--escalate-human-gate --ticket WP-XXXX`).
2. Sincronizar `STATE.md`, `TURN.md` y `execution_log.md` desde la misma
   autoridad canonica.
3. Hacer que `--mark-ready` y `--request-changes` lean el estado derivado del
   bus.
4. Validar con tests e2e que bus y proyecciones no vuelven a divergir.

## Invariantes

- El bus es la autoridad.
- `inspect`, `approve` y `changes` siguen la misma ruta de materializacion.
- No se introduce auto-healing silencioso.
- No hay import circular entre `bus/review_bridge.py` y `agent_controller.py`.
- El test roto `test_emit_fail_safe_on_review_decision` queda incluido en scope.
