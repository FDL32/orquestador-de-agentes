## Criterios de Aceptacion Global
- El guard bloquea `READY_FOR_REVIEW` si el arbol esta sucio o si falta el checkpoint M3.
- Las superficies vivas del runtime se excluyen para evitar falsos positivos.
- Los archivos fuera de `Files Likely Touched` se reportan como `scope_discrepancy` sin limpieza destructiva.
- `BUILDER_MILESTONE` deja trazabilidad de los checkpoints semanticos.
- El protocolo de recuperacion es literal, repetible y esta referenciado desde las reglas del Builder y la checklist del Manager.
- `HANDOFF_BLOCKED` lo emite `.agent/agent_controller.py` tras leer el JSON del guard, no el guard por si solo.

## Blockers
1. Si el guard no excluye las superficies vivas, cada handoff fallara por archivos que el propio runtime modifica.
2. Si M3 se auto-crea desde `--mark-ready`, el checkpoint deja de ser una ancla real y el guard pierde sentido.
3. Si el protocolo de recuperacion usa limpieza destructiva sobre archivos fuera de scope, se repite la perdida que origino WP-2026-165.
4. Si la base del diff de scope se calcula contra `HEAD~1`, el guard no protege el historial real del ticket.
5. Si `scope_discrepancy` se trata como bloqueo, el handoff se vuelve demasiado sensible y Builder no puede distinguir advertencia de falla.
6. Si el guard emite el evento directamente, se acopla innecesariamente al bus y complica la implementacion.
