# PLAN WP-2026-129 - Review env inheritance fix

## Contexto
WP-2026-129 corrige el entorno que el review bridge entrega a OpenCode para evitar que la herencia de `HOME`, `USERPROFILE` y `CODEX_HOME` se redirija a `.codex` y rompa el arranque del backend.

## Estrategia
Primero simplificar `_review_env()` para preservar el entorno del proceso y cambiar el fallback del Manager de `"codex"` a `"opencode"`. Despues validar que OpenCode y el review bridge siguen funcionando con el entorno heredado. Por ultimo asegurar el cambio con tests y quality gates.

## Alcance
- `bus/review_bridge.py` deja de redirigir `HOME`, `USERPROFILE` y `CODEX_HOME` a `.codex`.
- `bus/review_bridge.py` deja de usar `"codex"` como fallback del Manager y pasa a `"opencode"`.
- `tests/test_manager_review_bridge.py` cubre la herencia real del entorno de review.
- `tests/unit/test_review_env.py` valida el helper de entorno si se decide aislarlo en una prueba dedicada.
- `.agent/collaboration/work_plan.md`, `STATE.md`, `TURN.md` y `execution_log.md` quedan alineados con el nuevo ciclo.
- `PROJECT.md` refleja el nuevo ticket activo.

## Fases
1. Eliminar el redirect de home en `_review_env()` y dejar la copia del entorno tal como viene del proceso; cambiar el fallback del Manager a `"opencode"`.
2. Verificar el comportamiento de OpenCode con el entorno heredado y añadir tests de no-regresion.
3. Validar el ciclo completo con `ruff`, `pytest` y `--validate`.

## Entregable
Un review bridge que no encapsula OpenCode en una home artificial y mantiene el entorno del proceso intacto.

## Riesgos
- Romper compatibilidad con cualquier backend que dependiera del home aislado.
- Romper compatibilidad con flujos legacy que aún esperen `"codex"` como backend por defecto del Manager.
- Dejar un helper de review demasiado especifico para OpenCode si no se separa la logica por backend.
- Omitir tests de entorno y reintroducir el redirect por accidente.
