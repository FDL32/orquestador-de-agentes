## Objetivo
Hacer que `--mark-ready` falle cerrado si el Builder intenta cerrar un ticket sin un checkpoint M3 asociado al commit de entrega. El cierre debe exigir tanto el tag `checkpoint/review-<ticket>` como su asociacion al HEAD actual, con un mensaje explicito que le diga al Builder que cree el commit y el checkpoint antes de reintentar.

## Contexto
- `scripts/pre_handoff_guard.py` ya detecta falta de checkpoint M3 y arbol sucio, pero hoy solo comprueba que el tag exista.
- El Builder ha llegado varias veces al handoff sin consolidar commit + tag, y el guard solo lo bloquea al final.
- El cierre canonico debe convertirse en un contrato preventivo: no basta con tener el tag, el checkpoint debe corresponder al commit que se va a entregar.
- El mensaje de error debe ser accionable para Builder, no un bloqueo generico.

## Decision Arquitectonica
- `scripts/pre_handoff_guard.py` refactoriza `check_checkpoint_m3_exists()` en `check_checkpoint_alignment(project_root, ticket_id) -> tuple[bool, bool]`, que retorna `(missing_checkpoint, checkpoint_misaligned)`.
- `check_checkpoint_alignment()` reemplaza la funcion anterior, no se anade una segunda ruta de verificacion.
- La comparacion exacta usa `git rev-parse checkpoint/review-<ticket>^{}` contra `git rev-parse HEAD` del commit de entrega.
- El resultado del guard distingue `missing_checkpoint` de `checkpoint_misaligned` para que el mensaje de rechazo sea preciso.
- El mensaje de rechazo vive en `scripts/pre_handoff_guard.py`; `.agent/agent_controller.py` no necesita cambios si solo reenvia la salida del guard.
- El cierre no autocrea checkpoints ni tags desde `--mark-ready`.
- No se relaja la comprobacion de arbol limpio ni se amplia el contrato del guard con reintentos extra.

## Non-goals
- No auto-crear `checkpoint/review-<ticket>` desde `--mark-ready`.
- No cambiar el flujo de `create_checkpoint.py`.
- No relajar la deteccion de arbol sucio.
- No tocar el algoritmo de eventos del bus.

## Fases

### Fase 1: asociacion checkpoint-commit
- **Tipo:** TAREA AGENTE
- **Archivos:** `scripts/pre_handoff_guard.py`
- **Accion:** Modificar
- **Descripcion:** Refactorizar `check_checkpoint_m3_exists()` en `check_checkpoint_alignment(project_root, ticket_id) -> tuple[bool, bool]` para distinguir ausencia de tag de desalineacion. El guard debe usar `git rev-parse checkpoint/review-<ticket>^{}` contra `git rev-parse HEAD`. Añadir `checkpoint_misaligned: bool` al dict de resultado de `run_guard()` junto al `missing_checkpoint` existente, y reflejar ambos campos en la salida JSON del guard. El mensaje debe mostrar instrucciones exactas:
  - Tag ausente: `git commit && git tag -a checkpoint/review-<ticket> -m "Checkpoint M3 for <ticket>"`
  - Tag desalineado: `git tag -d checkpoint/review-<ticket> && git tag -a checkpoint/review-<ticket> -m "Checkpoint M3 for <ticket>"`
- **Riesgo:** Medio
- **Criterio de Aceptacion:** `--mark-ready` falla cerrado si falta el tag o si el tag no corresponde al commit de entrega, y la salida indica con claridad si debe crear el tag o recrearlo con el commit correcto.
- **Si falla:** Mantener el bloqueo por ausencia de tag y diferir la asociacion estricta a un ticket posterior.

### Fase 2: cobertura mecanica
- **Tipo:** TAREA AGENTE
- **Archivos:** `tests/test_pre_handoff_guard.py`
- **Accion:** Modificar
- **Descripcion:** Cubrir tres caminos: checkpoint M3 ausente, checkpoint presente pero desalineado con HEAD, y handoff valido con checkpoint asociado. Los tests deben verificar tambien el mensaje explicito de rechazo para Builder, las banderas separadas `missing_checkpoint` / `checkpoint_misaligned`, y las instrucciones exactas de commit/tag en stdout.
- **Riesgo:** Medio
- **Criterio de Aceptacion:** Los tests reproducen el rechazo por tag ausente, por tag desalineado y el camino feliz.
- **Si falla:** Conservar la validacion de tag existente y limitar el refuerzo a la capa de mensaje.
