# Plan de Trabajo - WP-2026-158

## Resumen
Este ticket endurece el review packet para que el Manager vea el alcance real del ticket, incluidos entregables nuevos no rastreados, y anade una metadata minima de filtrado/severidad inspirada en reviewdog.

## Objetivo
- Hacer visible cualquier deliverable nuevo que no aparezca en `git diff`.
- Mantener el diff como evidencia principal, pero no como unica fuente.
- Anadir una metadata minima de `filter_mode` y `severity` al packet.

## Reglas de diseno
- No cambiar `APPROVE / CHANGES / INSPECT`.
- No depender de `agent_controller` desde `bus/review_bridge.py`.
- No clonar reviewdog 1:1.
- No anadir dependencias nuevas.

## Fases
### Fase 1
- `bus/review_bridge.py` debe crear `._prepare_review_packet()` como helper nuevo de `_build_review_prompt()`; no existe aun.
- Ese helper incorpora la seccion explicita `--- Untracked Deliverables ---`.
- La seccion se inserta despues de `git diff` y antes del bloque final de instrucciones, para que el Manager la lea como evidencia contextual y no como decision.
- La fuente de `??` se obtiene con `git status --porcelain -z` sobre el repo raiz.
- Filtrado de ruido: excluir cualquier untracked bajo `.agent/collaboration/`, `.agent/runtime/`, `.git/` o artefactos generados equivalentes; incluir solo deliverables reales del ticket.
- Los tests demuestran que un repo temporal con `??` aparece en el packet.

### Fase 2
- `bus/review_bridge.py` publica `filter_mode` y `severity` minimos.
- Semantica de `filter_mode`:
  - `diff_context` = valor por defecto cuando el packet solo usa diff rastreado y provenance.
  - `added` = cuando el packet incluye uno o mas untracked deliverables elegibles.
  - `nofilter` = modo de respaldo reservado si el packet no puede aplicar filtrado; no debe ser el camino normal de este ticket.
- Los tests cubren un packet con solo diff rastreado y otro con entregables no rastreados.

## Archivos likely touched
- `bus/review_bridge.py`
- `tests/test_manager_review_bridge.py`
- `tests/test_review_bridge.py`

## Criterios de aceptacion
- El packet ya no oculta entregables nuevos no rastreados.
- La metadata de filtrado existe y no rompe el contrato de decision.
- Los tests pasan en aislamiento y en la suite safe.
