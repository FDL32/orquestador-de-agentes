# PLAN_WP-2026-146 - Human gate timeout and expiry

## Goal

Wire `HUMAN_GATE` escalation into the existing persistent approval subsystem so timeout handling survives restarts and reuses the approval-resolution contract already understood by the bus.

## Strategy

1. Add explicit timeout metadata to the `STATE_CHANGED -> HUMAN_GATE` handoff so the waiting window is visible in the bus payload.
2. Create and persist an `ApprovalRequest` in `bus/approval.py` when the ticket is escalated to `HUMAN_GATE`.
3. Reuse the supervisor's existing `check_and_expire_all()` loop to expire the persisted request and resolve the ticket canonically.
4. Keep the existing `APPROVAL_RESOLVED` -> `BLOCKED` path as the expiry contract, rather than introducing a new terminal state.
5. Add focused unit tests that simulate expired, non-expired, and restart-survivability cases.
6. Verify the change does not alter the existing `resume-human-gate` path or builder relaunch behavior.

## Scope Notes

- This ticket is about wiring the existing approval store, not a broader workflow redesign.
- No new ticket state is introduced; the timeout uses the existing approval resolution contract.
- `TURN.md` remains controller-owned and is not part of this change.
- The change should stay small: controller handoff, supervisor expiry, and one focused test file.
- `agents.json` should be the configuration source for the timeout value, with fallback documentation if the setting is missing.

## Acceptance Shape

- `HUMAN_GATE` events carry timeout metadata.
- Escalation persists an `ApprovalRequest` so the gate survives restarts.
- Expired `HUMAN_GATE` tickets resolve automatically through the canonical expiry path.
- Fresh `HUMAN_GATE` tickets remain open until their timeout is reached.
- The timeout value comes from configuration with a documented fallback.
- Existing `resume-human-gate` behavior remains unchanged.
- Tests cover expired, non-expired, and no-regression scenarios.
