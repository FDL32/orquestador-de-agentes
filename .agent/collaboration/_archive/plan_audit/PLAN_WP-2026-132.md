# PLAN_WP-2026-132 - Session-close observations for self-improving memory

## Objetivo
Definir una skill compartida de cierre de sesion que produzca observaciones curadas y reutilizables al final de cada ciclo, para que el sistema acumule aprendizaje estable sin contaminar `MEMORY.md` con ruido.

## Alcance
- `scripts/session_close_observations.py`
- `tests/unit/test_session_close_observations.py`
- `skills/session-close-observations/SKILL.md`
- `skills/session-close-observations/references/schema.md`
- `skills/session-close-observations/references/filter-rules.md`
- `skills/project-finalize/SKILL.md`
- `skills/README.md`
- `.agent/collaboration/work_plan.md`
- `.agent/collaboration/execution_log.md`
- `.agent/collaboration/STATE.md`
- `.agent/collaboration/TURN.md`

## Decision Arquitectonica
- `session-close-observations` es una skill compartida, usable por Builder, Manager o Supervisor al cerrar sesion.
- La helper script genera observaciones candidatas con un schema minimo y determinista.
- Las categorias validas son `convention`, `decision`, `fact` y `pattern`.
- Las reglas de filtrado son simples y conservadoras: hecho, persistencia entre sesiones y utilidad repetida.
- `stable: bool` no forma parte del schema; la estabilidad se infiere de los filtros y de la promocion posterior.
- `memory_consolidate.py` sigue siendo la capa de consolidacion; este ticket solo prepara la generacion de observaciones de cierre.
- `project-finalize` debe insertar la skill entre el Paso 9c (`local_audit.py`) y el Paso 9d (`memory_consolidate.py`).

## Fases
1. Crear el directorio de skill y el `SKILL.md` principal con workflow, limites y salida esperada.
2. Crear la helper script y los tests que validan schema, filtros y comportamiento de cierre.
3. Añadir referencias compactas para schema y reglas de filtro.
4. Actualizar `project-finalize` y `skills/README.md` para que el flujo de cierre exponga esta skill.
5. Ejecutar validaciones del catalogo de skills y del sistema completo.

## Calidad
- `python skills/validate_all.py`
- `python .agent/agent_controller.py --validate --json --force`

## Criterios de aceptacion
- La nueva skill y sus referencias existen y validan.
- La helper script produce observaciones candidatas con el schema acordado.
- Los tests cubren al menos: categoria invalida rechazada, campo requerido ausente, opinion/trace efimera filtrada y duplicado determinista.
- `skills/README.md` y `project-finalize` reflejan la nueva pieza del flujo.
- `validate_all.py` valida el frontmatter YAML de la nueva skill (`name`, `version`, `description`, `author`, `tags`), y el ticket no debe omitir el bloque `---`.
- No hay cambios en bus ni en state machine.
