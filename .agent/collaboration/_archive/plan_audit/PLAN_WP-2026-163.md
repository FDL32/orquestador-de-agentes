# Work Plan - WP-2026-163

## Metadata
- **ID:** WP-2026-163

## Objetivo
Definir la skill `man-session-closeout` y sus referencias para que el Manager pueda cerrar sesiones con una propuesta estructurada de learnings, clasificar cada aprendizaje como local, generalizable o dudoso, y dejar listo el puente humano hacia la mejora continua del motor sin meter automatizacion prematura.

## Contexto
- El repositorio ya aprende de codigo, review y calidad de tickets, pero no tiene un cierre de sesion que clasifique learnings por alcance.
- El nuevo flujo debe permitir que el agente proponga la clasificacion y que el usuario la valide antes de publicar nada en el canal compartido.
- La salida debe separar claramente learnings locales de los generalizables y dar un ciclo de vida a los dudosos con TTL de 3 WPs.
- La implementacion de este ticket es documental: primero se fija el contrato, luego se automatiza en un follow-up.

## Decision Arquitectonica
- La nueva skill sera `skills/man-session-closeout/SKILL.md` y funcionara como skill de Manager en stage `close`.
- El output del agente debe tener exactamente tres campos por item: `learning`, `categoria_propuesta`, `razon`.
- La validacion humana debe poder aceptar, recategorizar, descartar o diferir cada learning antes de persistirlo.
- `UPSTREAM_LEARNINGS.md` sera la cola humana de learnings generalizables y `closeout_lessons.md` sera el contexto puente para el siguiente plan.
- `dudoso` tendra TTL de 3 WPs y se archivara si no se reclasifica a tiempo.
- Este ticket no cablea aun la infraestructura de transporte ni los scripts de sincronizacion; solo deja el contrato escrito y consumible.

### Paso numerado en el handoff
`man-session-closeout` encaja como paso numerado entre `session-close-observations` y `memory-consolidate`:

1. `session-close-observations` genera observaciones de cierre.
2. `man-session-closeout` clasifica learnings por alcance y prepara el puente humano.
3. `memory-consolidate` consolida la memoria ya curada.
