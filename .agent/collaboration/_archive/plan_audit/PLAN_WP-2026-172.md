# PLAN - WP-2026-172

## Resumen
Este ticket endurece el flujo de cierre del Builder para evitar dos fallos recurrentes del ciclo anterior: `PROJECT.md` generando dirty_tree falso positivo y relanzado automatico del Builder tras `HANDOFF_BLOCKED`.

## Problema
- `PROJECT.md` se actualiza como parte del ciclo operativo, pero el guard aun puede tratarlo como cambio de handoff y bloquear el cierre si no esta en la lista de superficies vivas.
- `bus/supervisor.py` relanza Builder sin distinguir con suficiente precision entre un crash real y un bloqueo de contrato de entrega.
- El trigger de requeue puede aparecer antes del `HANDOFF_BLOCKED`; la supresion debe mirar eventos posteriores al trigger, no solo el ultimo evento visible.

## Objetivo
1. Tratar `PROJECT.md` como superficie viva tolerada por el pre-handoff guard.
2. Evitar el relanzado automatico del Builder cuando existe un `HANDOFF_BLOCKED` con `sequence_number > requeue_trigger_sequence`.

## Alcance
- `scripts/pre_handoff_guard.py`
- `bus/supervisor.py`
- `tests/test_pre_handoff_guard.py`
- `tests/test_supervisor.py`

## Decisiones
- `PROJECT.md` se añade a `LIVE_SURFACES_REL`.
- `run_once()` y `_bootstrap_requeue_if_needed()` deben suprimir el relanzado si existe cualquier `HANDOFF_BLOCKED` con `sequence_number > requeue_trigger_sequence`.
- El relanzado sigue permitido para timeout, lock stale o falta de evidencia de cierre.
- Cuando se suprime el relanzado, emitir `RELAUNCH_SUPPRESSED` con payload `{reason: "handoff_blocked", trigger_sequence: N, blocking_sequence: M}`.

## Criterios
- `PROJECT.md` no bloquea `--mark-ready` si es la unica diferencia relevante.
- `HANDOFF_BLOCKED` posterior al trigger de requeue no provoca relanzado automatico.
- Los tests cubren ambos comportamientos.
