# PLAN_WP-2026-131 - Memory index cap for persistent observations

## Objetivo
Limitar el crecimiento de `MEMORY.md` para que siga siendo un indice humano corto y estable, dejando `observations.jsonl` como la fuente historica de referencia.

## Alcance
- `scripts/memory_consolidate.py`
- `tests/unit/test_memory_consolidate.py`
- `.agent/runtime/memory/MEMORY.md`
- `AGENTS.md`
- `.agent/collaboration/work_plan.md`
- `.agent/collaboration/execution_log.md`

## Decision Arquitectonica
- `MEMORY.md` debe tener un tope explicito de 80 lineas.
- `observations.jsonl` conserva la historia completa y la busqueda profunda.
- `memory_consolidate.py` es la unica pieza que regenera el indice.
- `AGENTS.md` debe explicitar la regla para evitar drift documental.
- `memory_consolidate.py` debe exponer `MEMORY_MD_LINE_CAP = 80` y truncar el indice con un marcador visible si excede el limite.

## Fases
1. Introducir el cap de lineas para `MEMORY.md` en la regeneracion.
2. Ajustar el formato del indice para conservar solo senales de alto valor y truncar de forma honesta con marcador visible.
3. Añadir un test de cap que genere un indice demasiado largo y valide que el output final no supera 80 lineas.
4. Documentar la politica de memoria acotada en la seccion `Memoria por proyecto` de `AGENTS.md`.
5. Validar con tests y con `--validate` que no hay drift.

## Criterios de aceptacion
- `MEMORY.md` queda por debajo de 80 lineas.
- La historia completa sigue en `observations.jsonl`.
- El indice sigue siendo util para una sesion nueva.
- Las gates del ticket pasan sin tocar el bus ni el state machine.
