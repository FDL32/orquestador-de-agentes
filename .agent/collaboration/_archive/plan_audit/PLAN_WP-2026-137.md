# PLAN_WP-2026-137 - Supervisor startup lock and reconciliation dedupe

## Goal

Stop repeated `SUPERVISOR_RECONCILED` emissions when the supervisor or launcher is started multiple times on the same active ticket, without making the launcher the source of truth for safety.

## Strategy

1. Add an atomic supervisor instance lock so only one bootstrap path can own the active ticket at a time.
2. Make reconciliation idempotent for the same `previous_ticket` / `recovered_ticket` pair.
3. Keep the launcher non-blocking: it may warn or short-circuit cosmetically, but Python owns the real liveness decision.
4. Use a short fallback TTL when PID-based liveness is unavailable.
5. Add regression tests for repeated bootstrap, stale lock handling, duplicate reconciliation prevention, and launcher non-blocking behavior.

## Scope Notes

- This ticket is intentionally narrow.
- It does not change the observation pipeline, review memory, or the review rubric.
- Any broader launcher lifecycle rewrite should be split into a separate ticket.
- Semantics for "no changes" vs "lock rejected" must be documented in code, not inferred from a bare boolean.

## Acceptance Shape

- A repeated supervisor start for the same ticket does not emit duplicate `SUPERVISOR_RECONCILED` events.
- The lock acquisition is atomic and has no TOCTOU window.
- The launcher does not become the safety gate; Python handles liveness and lock rejection.
- The fallback TTL for stale-lock detection is short enough to avoid hour-long dead time.
- The recovered ticket remains on the correct active state.
- The tests prove the issue is fixed and the lock path is safe.
