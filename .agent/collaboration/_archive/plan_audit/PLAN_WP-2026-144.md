# PLAN_WP-2026-144 - Destination ticket prefix onboarding

## Goal

Make destination projects self-describing for ticket IDs. The installer should accept a local prefix, persist it into destination metadata, and the runtime should warn if a destination project has no declared namespace. The onboarding docs should point to the local `XXX-YYYY-NNN` convention.

## Strategy

1. Extend the installer so `--install --prefix XXX` can persist the destination ticket prefix.
2. Add the prefix to the destination link metadata so the namespace is auditable.
3. Teach `--validate` to warn when a host-project has no `Ticket prefix:` entry in `PROJECT.md`.
4. Align the bootstrap prompt and public docs with the destination namespace convention.
5. Extend tests to cover install, validation and metadata behavior.

Implementation note: `motor_destination_link.json` is generated in the destination workspace by the installer; the motor repo only changes the code that writes it, not the file itself.

## Scope Notes

- The motor repo keeps `WP-YYYY-NNN`.
- The destination repo uses `XXX-YYYY-NNN`.
- The new prefix is a local namespace contract, not a change to the bus schema.
- Existing state, memory and review flows stay intact.

## Acceptance Shape

- `--install --prefix XXX` writes the prefix into destination metadata.
- The destination link file stores the prefix for traceability.
- `--validate` warns when a destination omits its local prefix declaration.
- The bootstrap and repo docs describe the destination namespace consistently.
