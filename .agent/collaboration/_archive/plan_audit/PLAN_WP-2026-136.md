# PLAN_WP-2026-136 - Semantic --candidates input for session_close_observations

## Objetivo
Permitir que agentes externos, LLMs o usuarios aporten candidatos de observacion
ya construidos semanticamente a `session_close_observations.py` mediante un flag
`--candidates <json_file>`, desacoplando la generacion de candidatos de su
validacion y filtrado.

## Alcance
- `scripts/session_close_observations.py`
- `tests/unit/test_session_close_observations.py`
- `skills/session-close-observations/SKILL.md`

## Decision Arquitectonica
- `--candidates` y `--ticket` viven en un `add_mutually_exclusive_group(required=True)`.
  El contrato queda claro: una unica fuente de entrada por ejecucion, impuesto por
  argparse sin validacion manual en `main()`.
- `load_candidates_from_file(path: str) -> list[dict]` es el unico punto de entrada
  del modo candidates. Maneja de forma predecible y estricta:
    - Archivo ausente           -> `FileNotFoundError` con mensaje claro.
    - Bytes UTF-8 invalidos     -> `ValueError("UTF-8 decode error: ...")`
    - JSON invalido             -> `ValueError("JSON decode error: ...")`
    - Top-level no es lista     -> `ValueError("Expected list, got <type>")`
    - Elemento que no es dict   -> se salta con warning (no interrumpe el lote).
  No añade validaciones de schema: eso lo hace `validate_schema()` como antes.
- `load_existing_observations()` se endurece con `errors="replace"` para tolerar
  `UnicodeDecodeError` en `observations.jsonl`, igual que ya se hizo en
  `pre_compact_hook.py` (WP-2026-135). Un archivo danado no debe romper el cierre.
- `process_candidates()` y `append_observations()` no se tocan.
- stdlib only; sin dependencias nuevas.

## Files Likely Touched
- `scripts/session_close_observations.py`
- `tests/unit/test_session_close_observations.py`
- `skills/session-close-observations/SKILL.md`

## Fases
1. Endurecer `load_existing_observations()`: anadir `errors="replace"` al `open()`
   para sobrevivir bytes UTF-8 invalidos en `observations.jsonl`.
2. Implementar `load_candidates_from_file(path: str) -> list[dict]`:
   - `FileNotFoundError` si el archivo no existe.
   - `ValueError("UTF-8 decode error: ...")` si hay bytes invalidos.
   - `ValueError("JSON decode error: ...")` si el JSON esta roto.
   - `ValueError("Expected list, got <type>")` si el top-level no es lista.
   - Saltar con warning los elementos que no sean dict.
3. Refactorizar `argparse` en `main()`: mover `--ticket` y `--candidates` a un
   `add_mutually_exclusive_group(required=True)`. Eliminar cualquier validacion
   manual equivalente.
4. Ramificar en `main()`: si `args.candidates`, llamar `load_candidates_from_file()`;
   si `args.ticket`, llamar `extract_candidates_from_ticket()`.
5. Anadir tests en `test_session_close_observations.py`:
   - `--ticket` y `--candidates` son mutuamente excluyentes (argparse error).
   - `load_candidates_from_file()` con JSON valido -> devuelve lista de dicts.
   - `load_candidates_from_file()` con archivo ausente -> `FileNotFoundError`.
   - `load_candidates_from_file()` con JSON corrupto -> `ValueError`.
   - `load_candidates_from_file()` con top-level invalido (dict, string) -> `ValueError`.
   - `main()` con `--candidates` no llama a `extract_candidates_from_ticket()`
     (verificar dispatch con mock o patch).
6. Actualizar `skills/session-close-observations/SKILL.md`:
   - Comando de ejemplo con `--candidates`.
   - Workflow: nota de que la fuente puede ser `--ticket` o `--candidates`.
   - Nota explicita de exclusion mutua.

## Calidad
- `ruff check scripts/session_close_observations.py tests/unit/test_session_close_observations.py`
- `python -m pytest tests/unit/test_session_close_observations.py -v`
- `python scripts/session_close_observations.py --candidates <fichero.json> --dry-run --verbose`

## Criterios de aceptacion
- `--candidates <file>` funciona en modo dry-run sin errores.
- `--candidates + --ticket` a la vez produce error de argparse (exit != 0).
- Ninguna de las dos flags -> error de argparse (exit != 0).
- `load_candidates_from_file` lanza excepciones tipadas y predecibles para los
  cuatro casos de error.
- `load_existing_observations` tolera bytes UTF-8 invalidos sin romper el cierre.
- Los 6 tests nuevos pasan. `ruff` limpio.
