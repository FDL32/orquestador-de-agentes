# AUDIT_WP-2026-137 - Supervisor startup lock and reconciliation dedupe

## Audit Focus

Verify that the supervisor startup path is single-instance, that lock acquisition is atomic, and that reconciliation is idempotent for the same recovered ticket.

## What Must Be True

- Only one live supervisor instance can own the active ticket lock.
- Lock acquisition uses an atomic create path, not check-then-write.
- `SUPERVISOR_RECONCILED` is not re-emitted for the same `previous_ticket` / `recovered_ticket` pair.
- The launcher does not enforce safety by blocking on a file presence check.
- The launcher may be informational, but Python owns the real liveness and lock decision.
- If PID liveness is unavailable, the stale-lock fallback is short and does not freeze the system for an hour.
- Repeated bootstrap does not corrupt the ticket state.

## Failure Modes

1. A second supervisor instance starts while the first one is still active.
2. The same reconciliation payload is emitted more than once.
3. Lock acquisition has a TOCTOU window.
4. The launcher becomes the safety gate and blocks on stale locks forever.
5. A stale lock prevents a legitimate startup for too long.
6. Tests only cover the happy path and miss the duplicate-start regression.

## Review Checklist

- Confirm the lock path is canonical and cleaned up safely.
- Confirm the lock is acquired atomically.
- Confirm reconciliation dedupe keys the same logical recovery event.
- Confirm launcher behavior is non-destructive and non-blocking when the lock already exists.
- Confirm tests cover repeated startup, contended startup, stale locks, and launcher informational behavior.
