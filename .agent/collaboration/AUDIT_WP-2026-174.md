# Audit - WP-2026-174

## Lo que ya existe
- `scripts/manager_review_bridge.py` ya tiene una persistencia de heartbeat en `manager_bridge_state.json`.
- `tests/test_manager_review_bridge.py` ya tiene helpers para aislar el estado del bridge por test.
- `bus/supervisor.py` ya usa `manager_bridge_state.json` como señal de heartbeat, no como checkpoint duradero.
- El launcher de arranque ya puede limpiar `manager_bridge_state.json`, lo que refuerza la necesidad de separar el watermark en un archivo propio.

## Riesgos
- Si el watermark sigue mezclado con el heartbeat, un bootstrap puede repararlo o limpiarlo y volver a habilitar reprocesos.
- Si el checkpoint duradero no se carga antes de `_tick()`, el bridge puede revisar de nuevo un evento ya consumido.
- Si el artefacto nuevo no tolera ausencia o corrupcion, un reinicio del supervisor puede romper el bridge en lugar de recuperarlo.
- Si el arranque no toma el maximo entre ambas superficies, puede reaparecer el replay de eventos viejos.
- Si el bridge deja de escribir `last_processed_sequence` en `manager_bridge_state.json`, la compatibilidad durante la transicion se pierde.

## TP Check
- TP-01: El cambio es pequeño y acotado a persistencia del bridge.
- TP-02: No se toca `bus/supervisor.py`.
- TP-03: El checkpoint duradero queda separado del heartbeat.
- TP-04: Los tests cubren persistencia, restauracion, ausencia/corrupcion y el caso defensivo de merge por maximo.
- TP-05: La verificacion final usa `ruff`, `pytest` y `validate`.
