## Estado Actual
- La fase documental ya existe en WP-2026-161.
- Este ticket automatiza la deteccion mecanica de defectos de ticket y la expone en `--validate`.
- El objetivo es warning-only para no bloquear el flujo normal.

## TP Check
- TP-01: verificado con una secuencia unica sin pasos contradictorios en el audit.
- TP-02: verificado con criterios que citan comando o test literal.
- TP-03: verificado con Files Likely Touched concretos y sin comodines ambiguos.
- TP-04: verificado sin "si procede", "preferiblemente" ni lenguaje blando en el flujo critico.
- TP-05: verificado porque plan y audit describen la misma secuencia, archivos y resultado observable.

## Non-goals
- No `plan_accuracy_check.py`.
- No metricas F1 / precision / recall.
- No inyeccion programatica de observaciones en prompts.
- No bloqueo por warnings de prosa.

## Decision Arquitectonica
- Validador standalone en `scripts/validate_ticket_prose.py`.
- Integracion de warnings en `_handle_validate()`.
- Exit code 0 si solo hay warnings.
- Cobertura de unidad y de integracion para plan limpio y defectuoso.

## Blockers
1. `scripts/validate_ticket_prose.py` no existe o no detecta de forma estable los patrones acordados.
2. `.agent/agent_controller.py --validate` no agrega los warnings bajo `warnings.ticket_prose` o cambia la semantica de exit code.
3. No hay tests que cubran un plan defectuoso y uno limpio para validar el camino warning-only.
4. El validador introduce dependencias externas, muta el repo o trata los warnings como errores.
5. El validador no comprueba que `AUDIT_WP-*.md` existe y contiene `## TP Check`; sin esta comprobacion, el error que origino este ticket (TP Check omitido en el handoff) puede repetirse sin aviso automatico.

## Evidencia Esperada
- `python scripts/validate_ticket_prose.py` debe imprimir warnings de reglas detectadas sobre un plan defectuoso y salir con 0.
- `python scripts/validate_ticket_prose.py` debe emitir `audit-missing-tp-check` cuando no existe `AUDIT_WP-*.md` o no contiene `## TP Check`.
- `python .agent/agent_controller.py --validate --json --force` debe incluir la clave `warnings.ticket_prose` cuando el `work_plan.md` activo contenga patrones detectables.
- `python -m pytest tests/test_validate_ticket_prose.py tests/test_agent_controller.py -q` debe cubrir: plan limpio, plan defectuoso, AUDIT sin TP Check, y cada funcion de deteccion invocada directamente.

## Criterios de Aceptacion Global
- [ ] El validador standalone reporta warnings con regla, evidencia y sugerencia.
- [ ] El validador emite `audit-missing-tp-check` cuando el AUDIT carece de `## TP Check`.
- [ ] `--validate` expone warnings de ticket prose sin convertirlos en errores.
- [ ] Los tests cubren el caso limpio, el caso defectuoso y el AUDIT sin TP Check.
- [ ] Cada funcion de deteccion tiene test directo positivo y negativo.
- [ ] El ticket no cambia el comportamiento de `--force` mas alla de agregar warnings.
