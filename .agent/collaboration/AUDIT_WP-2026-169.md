# Audit - WP-2026-169

## Lo que ya existe
- `scripts/session_closeout.py` ya resuelve el cierre de sesion como orquestador standalone.
- `agent_controller.py` ya maneja `--validate`, `--mark-ready` y `--manager-approve`, asi que el nuevo flag debe integrarse en el mismo contrato de CLI.
- `tests/test_agent_controller.py` existe en el arbol, asi que el trabajo de test es de modificacion, no de creacion.

## Blockers
- Si `--session-close` reimplementa la pipeline de cierre en vez de delegar, el controlador duplicaria logica y quedaria fragil.
- Si el cierre real no sincroniza `STATE.md` y las proyecciones, el siguiente ciclo arrancaria con drift.
- Si `--dry-run` muta archivos reales, el wrapper romperia el contrato de cierre seguro.
- Si los tests no cubren delegacion, dry-run e idempotencia terminal, el entrypoint quedaria sin garantia mecanica.

## Evidencia esperada
- Un test de CLI que pruebe `--session-close` con `--dry-run`.
- Un test de cierre real que deje `STATE.md` alineado.
- Un test de idempotencia cuando `STATE.md` ya es terminal.
- Documentacion actualizada para que el entrypoint canonico sea visible.
