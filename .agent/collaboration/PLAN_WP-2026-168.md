# Plan - WP-2026-168

## Metadata
- **ID:** WP-2026-168
- **Estado:** APPROVED
- **deliverable_type:** code
- **Titulo:** Session closeout orchestrator - audit, memory, archive
- **Asignado a:** Builder

## Objetivo
Construir un orquestador de cierre de sesion en `scripts/session_closeout.py` que encadene el cierre tecnico de la sesion: auditoria del trabajo realizado, validacion de calidad, consolidacion de memoria, archivado de historial y comprobaciones de portabilidad. Este WP-A entrega solo el script y sus tests; la integracion con `agent_controller.py`, el template de cierre y la documentacion de protocolo quedan para WP-B.

## Contexto
- Hoy el cierre de sesion esta fragmentado entre `session_close_observations.py`, `memory_consolidate.py`, `archive_collaboration_artifacts.py`, `archive_execution_log.py`, `archive_event_bus.py`, `prepush_check.py` y `local_audit.py`.
- El cierre debe ser repetible y menos dependiente de memoria humana: el Builder debe tener un unico comando para ejecutar el cierre y obtener un reporte.
- La primera ejecucion no dispone de reporte previo; el rango de sesion debe arrancar desde el primer evento de `events.jsonl` salvo que se indiquen tickets explicitamente.
- La fuente de tickets sigue esta prioridad: tickets explicitos en CLI, tickets detectados en la ventana de sesion, y como fallback el ticket activo del `work_plan.md`.
- El objetivo es auditar el codigo y los tickets creados durante la sesion, pero sin invocar skills de prosa como si fueran scripts ejecutables.
- `code-audit` y `bui-self-audit` quedan como recomendaciones manuales en el reporte, no como dependencias ejecutables de WP-A.

## Decision Arquitectonica
- `scripts/session_closeout.py` es un orquestador standalone, invocable con `--project-root <path>`, `--dry-run`, `--skip-slow` y `--ticket/--tickets`.
- La ventana de sesion toma como base el timestamp del `session_close_report.md` mas reciente en `.agent/runtime/memory/`; cuando no existe un reporte previo, usa el primer evento de `events.jsonl`.
- La fuente de tickets sigue esta prioridad: tickets explicitos en CLI, tickets detectados en la ventana de sesion, ticket activo del `work_plan.md` si no hay eventos suficientes.
- El script llama a scripts reales mediante subprocess; no intenta ejecutar skills como `code-audit` o `bui-self-audit`.
- `prepush_check.py` es el gate bloqueante de calidad.
- `local_audit.py` es una auditoria informativa del estado del repo.
- `session_close_observations.py` se invoca una vez por ticket resuelto, en orden cronologico por timestamp de cierre; `memory_consolidate.py` se puede omitir con `--skip-slow`.
- El orquestador archiva historial con `archive_collaboration_artifacts.py`, `archive_execution_log.py` y `archive_event_bus.py --all-terminal`.
- La portabilidad se verifica con `git status --short`, `MANIFEST.distribute`, y una busqueda de rutas absolutas del workspace derivadas dinamicamente desde `Path.home()` y `Path(project_root).resolve()` en docs/ markdowns/skills/rules.
- El script escribe `.agent/runtime/memory/session_close_report.md` con el resultado de cada paso y con el resumen de la sesion.

## Non-goals
- No integrar `--session-close` en `.agent/agent_controller.py` en este WP.
- No crear `closeout_lessons.md`; ese puente queda para WP-B.
- No convertir skills de cierre en scripts ejecutables.
- No tocar el comportamiento de `session_close_observations.py`, `memory_consolidate.py` ni los archivadores existentes.
- No cambiar el ciclo canonico de tickets.

## Fases

- **Accion:** Crear
- **Descripcion:** Crear el orquestador de cierre con CLI propia. Debe resolver el rango de sesion con prioridad clara: tickets explicitos, tickets detectados en la ventana de eventos, y como fallback el ticket activo del `work_plan.md`. Si no existe reporte previo, la ventana arranca desde el primer evento de `events.jsonl`. El script debe producir `.agent/runtime/memory/session_close_report.md` aun en `--dry-run`.

### Fase 2: auditoria y calidad
- **Tipo:** TAREA AGENTE
- **Archivos:** `scripts/session_closeout.py`, `tests/test_session_closeout.py`
- **Accion:** Modificar
- **Descripcion:** Encadenar los gates de calidad y auditoria del cierre. `prepush_check.py` bloquea el cierre si falla. `local_audit.py` se ejecuta como snapshot informativo. `validate_ticket_prose.py --json` se incluye como verificacion de los tickets tocados durante la sesion. Si la sesion toco mucho codigo Python, el reporte debe recomendar manualmente `code-audit` y `bui-self-audit`, sin intentar ejecutarlos como comandos.
- **Riesgo:** Medio
- **Criterio de Aceptacion:** `prepush_check.py` es bloqueante; `local_audit.py` y `validate_ticket_prose.py` quedan reflejados en el reporte; el reporte distingue recomendaciones manuales de pasos automatizados.
- **Si falla:** Conservar solo el preflight bloqueante y dejar el resto como advertencias.

### Fase 3: memoria y archivado
- **Tipo:** TAREA AGENTE
- **Archivos:** `scripts/session_closeout.py`, `tests/test_session_closeout.py`
- **Accion:** Modificar
- **Descripcion:** Encadenar `session_close_observations.py` una vez por cada ticket resuelto, en orden cronologico por timestamp de cierre, y `memory_consolidate.py --verbose` salvo que se pase `--skip-slow`. El script debe archivar colaboracion, execution log y bus terminal con los archivadores existentes, y dejar reflejado en el reporte si algun paso fue omitido por `--skip-slow`.
- **Riesgo:** Medio
- **Criterio de Aceptacion:** La memoria se consolida cuando no se omite; `--skip-slow` salta observaciones y consolidacion sin romper el cierre; los archivadores dejan su resultado reflejado en el reporte.
- **Si falla:** Mantener el archivo manual de memoria y archivado como paso separado.

### Fase 4: portabilidad y reporte final
- **Tipo:** TAREA AGENTE
- **Archivos:** `scripts/session_closeout.py`, `tests/test_session_closeout.py`
- **Accion:** Modificar
- **Descripcion:** Verificar que `MANIFEST.distribute` existe, que `git status --short` queda limpio tras el cierre salvo por `.agent/runtime/memory/session_close_report.md`, y que no hay rutas absolutas del workspace local en docs/, markdowns, skills/ o `.agent/rules/`. Las coincidencias de rutas absolutas deben registrarse como warning no bloqueante. El script escribe `.agent/runtime/memory/session_close_report.md` con PASS/WARN/FAIL por paso y un resumen final del cierre.
- **Riesgo:** Bajo
- **Criterio de Aceptacion:** `MANIFEST.distribute` se valida, `git status --short` limpio se exige al final, las rutas absolutas solo generan warnings, y el reporte se genera en cada ejecucion.
- **Si falla:** Mantener la verificacion de portabilidad como chequeo manual.

## Criterios de aceptacion
- El orquestador resuelve el rango de sesion de forma determinista, incluso en la primera ejecucion.
- `--skip-slow` omite memoria/observaciones pero mantiene el cierre utilizable.
- `prepush_check.py` bloquea el cierre si la calidad no pasa.
- `validate_ticket_prose.py --json` valida los tickets tocados durante la sesion.
- `.agent/runtime/memory/session_close_report.md` resume auditoria, calidad, memoria, archivado y portabilidad con PASS/WARN/FAIL.
- El script no ejecuta skills como si fueran binarios; solo orquesta scripts reales.
- `git status --short` queda limpio al final del cierre automatizado salvo por el reporte runtime esperado en `.agent/runtime/memory/session_close_report.md`.

## Blockers
1. Sin una politica clara de rango de sesion, el cierre no sabe que tickets auditar.
2. Sin `--skip-slow`, el cierre puede volverse demasiado pesado para sesiones largas o urgentes.
3. Si el script intenta ejecutar skills de prosa como comandos, el flujo falla por diseno.
4. Sin verificacion concreta de `MANIFEST.distribute` y de rutas absolutas, la portabilidad queda solo declarativa.
5. Si el cierre bloquea por pasos informativos, se pierde utilidad operacional.
