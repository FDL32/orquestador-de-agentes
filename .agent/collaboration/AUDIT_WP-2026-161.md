# Audit - WP-2026-161

## Lo que ya existe
- El sistema ya tiene un catalogo canonico de anti-patrones de codigo (`AP-01..AP-14`) y checklists para Builder y Manager.
- Las guias de prompting refuerzan instrucciones directas, contexto explicito y ejemplos concretos; eso encaja con el formato TP que se quiere introducir.
- En WP-2026-160 quedaron tres defectos de ticket suficientemente claros como para convertirse en observaciones reutilizables.

## Blockers
- No existe un catalogo compartido de anti-patrones de ticket -> el ticket debe crear `skills/_shared/ticket-anti-patterns.md` con los TP de referencia.
- No existe una checklist de calidad de plan con preguntas verificables -> el ticket debe crear `skills/man-create-work-plan/references/plan-quality-checklist.md`.
- La skill del Manager no exige `## TP Check` en el audit -> el ticket debe actualizar `skills/man-create-work-plan/SKILL.md` para que la seccion sea obligatoria.
- Las observaciones de calidad de ticket no estan registradas de forma estructurada -> el ticket debe anadir tres observaciones ancladas en WP-2026-160 a `.agent/runtime/memory/observations.jsonl`.

## Evidencia esperada
- `skills/_shared/ticket-anti-patterns.md` con TP-01..TP-05, cada uno con descripcion, WHY, senal de deteccion y ejemplo `NO/SI`.
- `skills/man-create-work-plan/references/plan-quality-checklist.md` con preguntas verificables y una seccion `TP Check`.
- `skills/man-create-work-plan/SKILL.md` actualizada para exigir `## TP Check` en el `AUDIT_WP`.
- Tres nuevas observaciones en `.agent/runtime/memory/observations.jsonl` con `topic` de calidad de ticket y campos canonicos completos.
- `python scripts/validate_observations.py` y `python .agent/agent_controller.py --validate --json --force` verdes.
